This project manages JavaScript/TypeScript dependencies through Bun as the primary package manager, while keeping a package-lock.json (npm v3) alongside the canonical bun.lock. All third-party packages are declared in the root package.json under dependencies and devDependencies; there is no monorepo workspace — this is a single-package TanStack Start + Nitro SSR application.

### What system/approach is used
- Package manager: Bun (bun install, bun run ...). The bunfig.toml configures Bun's installer behavior globally for the repo.
- Manifest: package.json (ESM type module, private true, sideEffects false).
- Lockfiles: Both bun.lock (canonical, generated by Bun) and package-lock.json (npm v3 lockfile) are committed. They mirror the same dependency set but differ in format and sometimes version resolution; CI should pin on one tool to avoid drift.
- No vendoring or private registry: No vendor/, .npmrc, or custom registries are present. All packages resolve from the public npm registry.

### Key files and packages
- package.json — source of truth for declared versions (caret ranges on all deps).
- bun.lock — canonical deterministic lockfile consumed by Bun at install/build time.
- package-lock.json — secondary npm lockfile kept for compatibility (e.g. CI environments that default to npm).
- bunfig.toml — Bun configuration, notably the supply-chain safety policy.

### Architecture and conventions
- Versioning strategy: All entries use caret (^) ranges, allowing minor/patch upgrades within the major version. There is no explicit policy restricting which ranges may be used.
- Supply-chain guard: bunfig.toml sets minimumReleaseAge = 86400 (24 hours), blocking installation of any package published less than a day ago. Packages exempted via minimumReleaseAgeExcludes must be reviewed before adding — currently only Lovable dev tooling (@lovable.dev/vite-tanstack-config, @lovable.dev/mcp-js) is whitelisted.
- Runtime vs build-time split: Heavy UI/runtime libraries live in dependencies (React 19, Radix primitives, Tailwind v4, TanStack Router/Start, Zod, Recharts, etc.); tooling (Vite, ESLint, Prettier, Playwright, TypeScript, Nitro beta) lives in devDependencies.
- Build pipeline integration: Scripts delegate to Vite (vite dev, vite build); the check script chains lint -> typecheck -> test -> build using bun run, tying dependency availability directly into developer workflows.

### Rules developers should follow
1. Always use Bun for installs and scripts (bun install, bun run <script>). Do not switch to npm/yarn/pnpm locally or in CI without updating both lockfiles consistently.
2. Add new packages via bun add / bun add -D so bun.lock stays authoritative. If you update package.json manually, regenerate the lockfile with bun install.
3. Respect the 24-hour release-age guard. New packages will fail to install until they age out; if you must bypass it, add the package to minimumReleaseAgeExcludes in bunfig.toml and document why.
4. Keep package-lock.json in sync when switching between Bun and npm in CI. Prefer standardizing on one resolver to avoid subtle version divergence.
5. Do not introduce a private registry or .npmrc unless explicitly approved — the current setup assumes the public npm registry only.
